# agentome

> A self-hosted MCP server that gives LLM agents accurate, version-matched API documentation for Python libraries.

`agentome` sits between your agent and your internal library documentation. It exposes a small set of MCP tools that let an agent discover what packages are available, check what versions have artifacts, and retrieve API references — down to individual classes and functions — matched exactly to the version installed in the project it is working on.

```
Agent: "What version of data-pipeline is installed?"
  → importlib.metadata → "1.8.0"
Agent: calls get_symbol("data-pipeline", "1.8.0", "NormalizationPipeline")
  → agentome returns accurate signatures and docstrings
Agent: writes correct code on the first attempt
```

Artifacts are generated by [`llmscribe`](https://github.com/your-org/llmscribe) and stored as plain JSON on the local filesystem.

---

## Why

Internal Python libraries are invisible to LLMs — they have zero training data on them. Agents working with private codebases guess at API signatures, invent keyword arguments, and hallucinate class hierarchies. The problem gets worse when different projects pin different versions of the same internal library.

`agentome` solves the **serving** side of this:

- Runs locally on developer machines or on a shared internal server
- Serves versioned API artifacts generated by [llmscribe](https://github.com/kschweiger/llmscribe)
- Exposes a clean MCP interface that any MCP-compatible agent can use
- Zero cloud dependency, zero vendor lock-in

```bash
pip install agentome
```

## Setup

### 1. Generate artifacts with llmscribe

First, generate API artifacts for your internal libraries using [`llmscribe`](https://github.com/your-org/llmscribe). This is typically done in CI on each release:

```bash
pip install llmscribe
llmscribe generate your-data-pipeline --out ./artifacts
llmscribe generate your-auth-lib --out ./artifacts
```

This produces:

```
artifacts/
├── your-data-pipeline/
│   ├── 1.7.0.json
│   └── 1.8.0.json
└── your-auth-lib/
    └── 1.0.0.json
```

### 2. Start agentome

```bash
agentome --artifacts ./artifacts
```

### 3. Configure your MCP client

In opencode

```json
{
  "$schema": "https://opencode.ai/config.json",
  "mcp": {
    "agentome": {
      "type": "remote",
      "url": "http://localhost:8000/mcp",
      "enabled": true,
    }
  }
}
```

## MCP Tools

Once running, `agentome` exposes four tools to the agent:

### `list_packages()`

Discover what packages have artifacts available on this server.

```json
{
  "packages": {
    "your-data-pipeline": ["1.7.0", "1.8.0"],
    "your-auth-lib": ["1.0.0"]
  }
}
```

### `list_versions(package)`

Check what versions are available for a specific package.

```json
{
  "package": "your-data-pipeline",
  "versions": ["1.7.0", "1.8.0"],
  "latest": "1.8.0"
}
```

### `get_symbol(package, version, symbol)`

Retrieve the API reference for a single class or function. **Prefer this over `get_api()`** — it is more token-efficient and gives the agent exactly what it needs.

```json
{
  "package": "your-data-pipeline",
  "version": "1.8.0",
  "symbol": "NormalizationPipeline",
  "api": {
    "kind": "class",
    "docstring": "Chains normalization transformations over a dataset.",
    "members": {
      "run": {
        "kind": "function",
        "parameters": [
          { "name": "data", "type": "pd.DataFrame" },
          { "name": "strict", "type": "bool", "default": "True" }
        ],
        "returns": "pd.DataFrame",
        "docstring": "Execute the pipeline."
      }
    }
  }
}
```

### `get_api(package, version)`

Return the full API reference for an entire package version. Use this when the agent needs a broad overview rather than a specific symbol.

## Transports

`agentome` supports both MCP transports:

```bash
# stdio (default) — for local MCP clients like Claude Desktop and Claude Code
agentome --artifacts ./artifacts

# HTTP — for shared internal servers or remote access
agentome --artifacts ./artifacts --transport http --port 8000
```

---

## Configuration

| Flag | Env var | Default | Description |
|---|---|---|---|
| `--artifacts` | `AGENTOME_ARTIFACTS_DIR` | - | Path to artifacts directory |
| `--transport` | — | `stdio` | `stdio` or `http` |
| `--port` | — | `8000` | Port for HTTP transport |

## How Versioning Works

The agent is responsible for determining which version to request — `agentome` is a dumb artifact store. The recommended pattern:

```python
# The agent runs this in the target environment
import importlib.metadata
version = importlib.metadata.version("your-data-pipeline")  # → "1.8.0"
# Then calls: get_symbol("your-data-pipeline", "1.8.0", "MyClass")
```

This keeps version resolution in the environment that actually has the correct answer, rather than guessing from the server side.

## Relation to llmscribe

```
your CI pipeline                    developer machine / internal server
─────────────────────               ────────────────────────────────────
llmscribe generate mylib            agentome --artifacts ./artifacts
  └── artifacts/                      └── MCP server
      └── mylib/                          ├── list_packages()
          └── 1.8.0.json                  ├── list_versions("mylib")
                                          ├── get_api("mylib", "1.8.0")
                                          └── get_symbol("mylib", "1.8.0", "MyClass")
```

`agentome` and `llmscribe` are intentionally decoupled — you can point `agentome` at artifacts generated by a different tool, or use `llmscribe` artifacts with a different server. The artifact format is plain JSON with no `agentome`-specific envelope.
